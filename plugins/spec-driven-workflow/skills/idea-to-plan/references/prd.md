# PRD 템플릿 (1단계)

`planning/{slug}/prd.md`로 저장. 아래 5개 섹션·필드 구조를 정확히 따른다 (Manny 라이브 산출물 기준).
이 문서가 **제품 차원의 SOT** — 이후 모든 산출물이 여기서 유도된다.

## 정확한 구조

```markdown
# {제품명}

|  |  |
|---|---|
| **프로젝트 이름** | {제품명} |
| **날짜** | {YYYY-MM-DD} |

---

## 1. Overview (개요)

### Summary (한 줄 정의)
{제품을 한 문장으로}

### Product Goals (제품 목표)
{이 제품이 달성하려는 것. 측정 가능하게.}

### Context (배경)
{왜 지금 이게 필요한가 — 시장/사용자 맥락}

## 2. Problem & Solution (문제와 해결)

### User Problem (사용자 문제)
{누가, 어떤 상황에서, 무엇 때문에 곤란한가}

### Solution (해결 방안)
{제품이 그 문제를 어떻게 푸는가}

### Differentiator (차별점)
{경쟁/대안 대비 핵심 차별점 — 번호 목록}

## 3. Target & Scenario (타겟과 시나리오)

### Target Users (타겟 사용자)
{주 사용자는 누구이고 무엇을 중요시하는가}

### User Scenarios (사용자 시나리오)
{대표 사용자가 제품을 쓰는 1~7단계 내러티브 — 유저플로우의 씨앗}

## 4. Success & Risks (성공지표와 리스크)

### Key Metrics (핵심 지표)
{성공을 어떻게 측정하나 — 정량 지표}

### Risks (리스크)
{보안/의존성/성능/학습곡선/경쟁 등 주요 위험}

## 5. Project Basics (프로젝트 기본)

### Category (카테고리)
> {예: 생산성 / 업무}

### User Roles (사용자 역할)
> {예: 사용자, 관리자}   ← 기능명세서에서 각 기능의 역할로 재사용된다

### Devices (디바이스)
> {예: 웹, macOS 앱, 모바일 PWA}   ← 기능명세서에서 각 기능의 기기로 재사용된다
```

## 작성 규칙

- **User Roles와 Devices는 하류로 흘러간다.** 기능명세서가 각 기능에 다는 "역할/기기" 태그가 여기서 정의한 값이어야 한다. 신중히 확정한다.
- User Scenarios는 구체적 인물·동선으로 쓴다 ("개발자 김프로는 퇴근 전 …"). 이게 유저플로우 3단계의 직접 입력이 된다.
- 모호어("빠르게", "직관적") 대신 측정 가능한 표현. 측정 가능성은 Key Metrics에서 다시 확인.
- 비어 있는 필드를 추측으로 채우지 말 것 — 0단계로 돌아가 묻는다.

## 게이트

사용자가 PRD 전체를 읽고 "이대로 간다"에 동의해야 2단계로 간다. 동의 없이는 기능명세 0줄.
