# 인터뷰 방법론 (0단계)

목적은 **수렴**이다 — 막연한 아이디어를 "한 문장 정의 + 핵심 결정들"로 좁힌다. 설득이나 브레인스토밍 발산이 아니다.

## 어떻게 묻는가

- **선택형으로 묻는다.** 자유서술보다 단일/복수 선택이 답하기 쉽고 결정을 빠르게 만든다. `AskUserQuestion` 도구를 쓴다.
- **한 번에 3~4개**씩, 꼬리에 꼬리를 문다. Manny는 총 10개를 단계적으로 물었다 — 한 번에 다 쏟지 말 것.
- **추천안을 첫 옵션**으로 두고 근거를 단다. 사용자가 결정을 위임할 수 있게.
- 사용자가 이미 답한 것은 다시 묻지 않는다. 대화 맥락에서 먼저 추출하고, **빈 칸만** 묻는다.

## 무엇을 묻는가 (커버리지 체크리스트)

아래가 다 채워지면 PRD를 쓸 수 있다. Manny의 Q1~Q10이 이 항목들을 덮었다.

| # | 묻는 것 | PRD 어디로 가나 |
|---|---|---|
| 1 | 이 제품이 달성할 **단 하나의 목표** | Product Goals |
| 2 | 해결할 **사용자 문제**(복수 가능) | User Problem |
| 3 | **타겟 사용자**와 그들이 중요시하는 것 | Target Users |
| 4 | **핵심/차별 기능**(복수 가능) | Differentiator / 기능명세 시드 |
| 5 | 사용자가 **어떤 흐름**으로 쓸지 | User Scenarios / 유저플로우 시드 |
| 6 | 주로 **어떤 환경**에서 접속하나 | Devices |
| 7~9 | 도메인 특화 핵심 관심사 (예: 비용/권한/실시간성) | 해당 기능 요구사항 |
| 10 | **관리자/운영** 측면 필수 기능 | 관리자 역할 기능 |
| - | **기술 스택**(알면) | Project Basics / 와이어프레임 힌트 |

> 도메인에 맞게 7~10번은 바꾼다. 예: 핀테크면 규제/정산, 커머스면 결제/배송. "이 제품 특유의 위험한 미정 결정"을 묻는 게 핵심이다.

## 질문 작성 예시

```
Q. 이 제품이 가장 중요하게 달성하려는 목표는? (단일 선택)
  - (추천) 로컬 PC가 꺼져도 AI 에이전트 작업의 끊김 없는 연속성 보장
  - 여러 에이전트 작업을 한 화면에서 동시 모니터링
  - 작업 비용을 정확히 추적·통제
```

## 통과 게이트

- 제품을 **한 문장**으로 말할 수 있다.
- 위 체크리스트의 1~6번이 답해졌다 (7~10은 제품에 따라 선택).
- **후보 접근법 · 미해결 질문 · 비자명한 결정의 근거(결정 노트)가 정리됐다** — OpenSpec 단계 0(탐색)이 남기던 산출물을 여기서 흡수한다(이후 PRD의 배경·시나리오·리스크로 흘려보낸다).
- 충족되면 1단계(PRD)로. 안 되면 부족한 항목만 추가 질문.
