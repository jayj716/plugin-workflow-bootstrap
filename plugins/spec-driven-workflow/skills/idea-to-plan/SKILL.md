# Idea → Plan (제품기획 파이프라인)

막연한 아이디어 한 줄을 받아, **PRD → 기능명세서 → 유저플로우 → 와이어프레임**을 순서대로 만든다.
manyfast의 "Manny" 기획 흐름을 Claude Code에 옮긴 것이며, 에이전틱 코딩 워크플로우(OpenSpec=What /
Superpowers=How·TDD)에서 OpenSpec이 담당하는 "무엇을(What)"의 **더 앞단** — 제품 차원의 기획 — 을 책임진다.
(해당 워크플로우 문서가 프로젝트에 있으면 `workflows/Agentic-Coding-Workflow.md`를 참조한다.)

```
[아이디어] ──▶ 0.인터뷰 ──▶ 1.PRD ──▶ 2.기능명세서 ──▶ 3.유저플로우 ──▶ 4.와이어프레임
                                                                              │
                                                          5.OpenSpec 핸드오프 ─┘
                                                          (상세기능 → EARS → /opsx:propose)
```

## 핵심 원칙 (왜 이렇게 일하는가)

기획 산출물이 망가지는 단 하나의 이유는 **상류 결정이 바뀌었는데 하류 문서가 따라오지 못해서**다.
(Manny 대화 실제 사례: "단일 페이지 워크스페이스"로 요구가 바뀌자 PRD→유저플로우→와이어프레임을 전부 다시 맞춰야 했다.)
그래서 이 스킬은 4가지를 강제한다 — 이 저장소 워크플로우의 "스펙=SOT" 철학과 동일하다.

1. **PRD가 SOT다.** PRD가 제품 차원의 유일한 진실원. 기능명세·유저플로우·와이어프레임은 모두 PRD에서 *유도된* 산출물이다.
   상류가 바뀌면 코드(하류)를 손대기 전에 **상류부터 고치고 하류로 전파**한다.
2. **인터뷰 먼저.** 산출물을 쓰기 전에 부족한 정보는 **선택형 질문**으로 수렴시킨다. 추측으로 채우지 않는다 — 추측은 나중에 전파 비용으로 돌아온다.
3. **순서 게이트.** 앞 산출물을 사람이 확인하기 전에 다음 산출물로 넘어가지 않는다. 흐릿한 PRD 위에 정교한 기능명세를 쌓으면 둘 다 버린다.
4. **계층 일관성.** 기능명세서는 항상 `요구사항 > 기능 > 상세기능` 3계층. 상세기능 하나가 나중에 EARS 요구사항 1줄 → tasks.md 1줄 → RED 테스트 1개로 내려간다 (5단계).

## 산출물 위치

한 아이디어 = 한 폴더. 산출물이 곧 인터페이스다(워크플로우 철학과 동일).

```
planning/{slug}/
├── prd.md              # 1단계 — 제품 차원 SOT
├── feature-spec.md     # 2단계 — 요구사항>기능>상세기능
├── userflow.md         # 3단계 — Mermaid flowchart
└── wireframe.html      # 4단계 — 실제 열어볼 수 있는 HTML/CSS
```

`{slug}`는 제품명의 kebab-case (예: `relay-remote-agent`). 폴더가 없으면 만든다.

## 진행 절차

각 단계는 **트리거 → 인터뷰(필요시) → 산출물 작성 → 사람 확인(게이트) → 다음 단계**.
한 번에 다 만들지 말고, 단계마다 멈춰 사용자에게 확인받는다. 사용자가 "한 번에 다 해줘"라고 하면 그때 연속 진행한다.

### 0단계 — 인터뷰 (수렴)

아이디어가 한 문장으로 명확하지 않으면 먼저 질문한다. 코드도 산출물도 만들지 않는다.
질문 설계 방법(단일/복수 선택, 무엇을 물을지, 몇 개나)은 **[references/interview.md](references/interview.md)** 를 읽고 따른다.

- 가능하면 `AskUserQuestion` 도구로 선택형 질문을 제시한다(Manny의 Q1~Q10 방식).
- **탐색 노트를 남긴다 (explore 흡수):** 수렴하면서 *후보 접근법 · 미해결 질문 · 비자명한 결정의 근거(결정 노트)* 를 정리해 PRD의 배경·시나리오·리스크에 반영한다. 이것이 OpenSpec 단계 0(탐색)이 남기던 산출물을, **도구 의존 없이** 더 풍부하게 대신한다.
- 통과 기준: 제품을 **한 문장으로** 정의할 수 있고, 타겟·핵심기능·플랫폼이 답해졌으며, 후보안·미해결 질문이 정리됐다.
- 이미 충분한 맥락이 대화에 있으면 인터뷰를 건너뛰고 1단계로 간다.

> **이 0단계가 곧 에이전틱 코딩 워크플로우의 "단계 0 탐색"의 확장판이다.** idea-to-plan으로 기획을 마치면 탐색은 끝난 것으로 보고, 5단계에서 곧장 `/opsx:propose`(단계 1)로 넘어간다 — `/opsx:explore`를 따로 호출하지 않는다.
>
> 이 스킬은 Claude·Codex·Cursor·npx-skills 어디서나 동작해야 하므로 OpenSpec 전용 도구(`/opsx:*`)에 **의존하지 않는다**(인터뷰는 위 자체 절차로 수행). 제품 기획까진 필요 없이 작은 기술 변경 하나만 생각을 정리하려는 경우라면, idea-to-plan 대신 `/opsx:explore`(또는 `superpowers:brainstorming`)를 쓰는 게 더 맞다 — 둘은 형제 진입점이지 중첩이 아니다.

### 1단계 — PRD (제품 차원 SOT)

`planning/{slug}/prd.md` 작성. 형식·섹션 구성은 **[references/prd.md](references/prd.md)** 를 정확히 따른다
(Manny 라이브 산출물 기준 5개 섹션: Overview / Problem & Solution / Target & Scenario / Success & Risks / Project Basics).

- 게이트: 사용자가 PRD를 읽고 "이대로 간다"에 동의. 동의 전 2단계 금지.
- PRD가 바뀌면 이후 모든 하류 산출물에 **전파 필요** 표시(아래 "변경 전파" 참조).

### 2단계 — 기능명세서

`planning/{slug}/feature-spec.md` 작성. **[references/feature-spec.md](references/feature-spec.md)** 의 3계층 템플릿을 따른다.

- 모든 기능은 PRD의 목표·핵심기능에서 유도되어야 한다. PRD에 없는 기능이 생기면 그것은 **PRD 변경 사건** — 1단계로 올라가 PRD부터 고친다(원칙 1).
- 각 항목에 중요도(🔴/🟡/🟢)·역할·기기, 요구사항에는 수용 기준을 단다.
- 게이트: 사용자 확인 후 3단계.

### 3단계 — 유저플로우

`planning/{slug}/userflow.md` 작성. Mermaid `flowchart` + 영역별 `subgraph`. 규칙은 **[references/userflow.md](references/userflow.md)**.

- 유저플로우의 모든 화면/분기는 기능명세서의 기능과 대응되어야 한다. 빠진 기능·뜬 화면이 있으면 불일치 → 고친다.
- 게이트: 사용자 확인 후 4단계.

### 4단계 — 와이어프레임

`planning/{slug}/wireframe.html` 생성. 실제 브라우저로 열어볼 수 있는 HTML/CSS. 규칙은 **[references/wireframe.md](references/wireframe.md)**.

- 화면 구성은 유저플로우의 노드 ↔ 기능명세서의 기능과 1:1로 맞춘다.
- 고품질 UI가 필요하면 `frontend-design` 스킬을 함께 사용한다. 단, 여기 목적은 *완성 UI*가 아니라 *구조·배치 합의*다 — 저충실도(low-fidelity)로 빠르게.

### 5단계 — OpenSpec 핸드오프 (선택)

기획이 끝나면 구현 파이프라인으로 넘긴다. **[references/openspec-handoff.md](references/openspec-handoff.md)** 를 따라
기능명세서의 **상세기능 → EARS 요구사항**으로 변환하고 `/opsx:propose`(없으면 `openspec` CLI)로 연결한다.
이 단계는 사용자가 "이제 구현하자"라고 할 때만 수행한다.

## 변경 전파 (가장 중요)

상류 산출물이 바뀌면, 하류를 그대로 두지 않는다. 이것이 Manny가 한 핵심 동작이고 이 스킬의 존재 이유다.

- **PRD 변경** → 영향받는 기능명세·유저플로우·와이어프레임 항목을 짚어, *무엇을 어떻게* 바꿔야 하는지 사용자에게 제시하고 갱신한다.
- **기능명세 변경** → 유저플로우·와이어프레임의 해당 부분 갱신.
- **유저플로우 변경** → 와이어프레임 갱신.
- 절대 조용히 한쪽만 바꾸지 않는다. **상류만 고치고 멈추지 마라** — 알리는 것과 전파하는 것은 별개이며 둘 다 해야 한다.
  - 기본 동작: 영향받는 하류를 **모두 갱신한 뒤**, "이 변경을 하류 X·Y·Z까지 전파해 갱신했습니다"라고 보고한다.
  - 사용자가 이미 변경을 지시·승인한 상태면(예: "이렇게 바꿔줘") 되묻지 말고 바로 전파한다. 전파 자체를 질문으로 막지 않는다.
  - 전파 범위가 크거나 파괴적이어서 확인이 필요할 때만 "X·Y에 전파됩니다 — 진행할까요?"로 먼저 묻는다.
  - 즉, "갱신할까요?"는 멈춤 신호가 아니라, 큰 전파에 한해 쓰는 예외적 확인이다. 작은/명확한 변경은 알림과 함께 곧장 전파.

## 특정 산출물만 다시

전체 파이프라인을 돌 필요 없이 한 산출물만 재생성/수정해달라는 요청(예: "유저플로우만 다시")이면,
해당 단계의 reference만 읽고 그 산출물만 작업하되, **변경 전파** 규칙은 반드시 적용한다.
