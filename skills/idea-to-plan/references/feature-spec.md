# 기능명세서 템플릿 (2단계)

`planning/{slug}/feature-spec.md`로 저장. **`요구사항 > 기능 > 상세기능`** 3계층을 항상 유지한다 (Manny 라이브 산출물 기준).

## 3계층의 의미

| 계층 | 마크다운 | 역할 | 비유 |
|---|---|---|---|
| **요구사항** | `## N. 제목` | 큰 묶음. 수용 기준(Acceptance Criteria)을 가진다 | OpenSpec의 한 change |
| **기능** | `### N.M 제목` | 사용자가 인지하는 기능 단위. 역할/기기 태그 | 한 화면/한 동작 |
| **상세기능** | `#### N.M.K 제목` | 구현 단위. **이게 5단계에서 EARS 1줄·테스트 1개가 된다** | tasks.md 한 줄 |

## 정확한 구조

```markdown
# {제품명} — 기능명세서

> PRD: planning/{slug}/prd.md (이 문서의 상류 SOT)

## 1. {요구사항 제목}
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

{이 요구사항이 무엇을 보장하는지 1~2문단}

**수용 기준:**
```
1. ○ {검증 가능한 조건}
2. ○ {검증 가능한 조건}
```

### 1.1 {기능 제목}
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: 사용자 | **기기**: 웹, macOS 앱

{이 기능 설명. 연결 동선이 있으면 함께}

#### 1.1.1 {상세기능 제목}
{무엇을 구현하는지 — 한 가지 검증 단위}
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: 사용자 | **기기**: 웹, macOS 앱

#### 1.1.2 {상세기능 제목}
...
```

## 태그 규약

- **중요도**: 🔴 높음 / 🟡 중간 / 🟢 낮음
- **진행 상태**: ⚪ 작성중 / 🔵 진행중 / ✅ 완료
- **역할 / 기기**: PRD의 User Roles / Devices에서 정의한 값만 사용 (새 값이 필요하면 PRD를 먼저 고친다)

## 작성 규칙

- **모든 기능은 PRD에서 유도된다.** PRD의 Product Goals·Differentiator·핵심기능에 근거 없는 기능을 새로 만들면, 그건 PRD 변경 사건 → 1단계로 올라간다(원칙 1).
- **수용 기준은 요구사항(`##`) 레벨에만** 단다. 검증 가능하게, 결과 중심으로.
- **상세기능 = 하나의 검증 단위.** "A하고 B한다"처럼 두 개가 섞이면 쪼갠다. 5단계에서 EARS 한 줄로 깔끔히 내려가야 한다.
- **연결 동선**(이 기능 → 저 기능으로 이어지는 화면 흐름)이 있으면 기능 설명에 명시한다 — 유저플로우 3단계의 입력이 된다.

## 게이트

사용자 확인 후 3단계. 모든 요구사항에 수용 기준이 있고, 3계층이 일관되며, 역할/기기가 PRD와 일치하는지 점검.
